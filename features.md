## Suggested order

Topological build order per the design doc (§6).

1. `webapp-skeleton` — bones first; everything depends on it
2. `doc-discovery` — filesystem walker + SQLite index
3. `read-state` — stamp last_read_at on every view
4. `inbox-view` — the headline UX
5. `doc-view` — render individual docs in the webapp shell
6. `sse-refresh` — live inbox updates via SSE
7. `synthesis-response-capture` — POST instead of clipboard
8. `comment-capture` — persistent click-to-comment *(folded into `skill-webapp-integration`)*
9. `skill-webapp-integration` — drop Chrome + clipboard from feature-skills; read responses & comments over HTTP

| [versioned-content-store](docs/features/versioned-content-store/context.md) | **Shipped.** F1 of the structured-content arc: document content now lives in the DB as versioned, structured sections via a re-runnable importer. The walker (`walker.py`) parses context/requirements/plan into ordered sections (feedback & the tracker ride as opaque whole-bodies) using a new pure `doc_content.py` (manifest-driven, `convert_charrefs=False` for faithful bodies), and records a content snapshot in a new `document_versions` table — cutting a version only on real content change, so no-op re-saves stop surfacing in the inbox. Documents gained a path-independent `logical_key` (migration 0003 + an idempotent startup backfill that dedupes collisions, repointing read-state/synthesis-responses/comments onto the survivor) decoupling identity from `source_path` for the future agent-submission API; archival became a status change on one row. `content_html` was deliberately left NULL so rendering stays disk-based — the version accessor is the seam F2 will consume. Three phase PRs (#26–28) + one direct-to-main, test-only review round; 372 tests, ruff/ty clean. |

New readable pages — per-feature `/project/{p}/feature/{slug}` (status/owner/notes + docs grouped primary / feedback-with-awaiting-badge / archived) and per-project `/project/{p}` (features grouped In progress / Available / Done, plus the tracker doc) — keyed by name+slug with 404-on-unknown, URL-encoded links, and SSE live-refresh that suppresses the initial connect message to avoid a reload loop. Inbox cards were restructured into per-element links (per-document sections open the doc from the feature name; per-feature sections link to the feature page; project names link to the project page) and a "Home" breadcrumb root was added across the doc view and both new pages. Four phase PRs (#22–25) + one direct-to-main review-fix round (Proposal A inbox-link semantics, "Home" crumb, `_rank`→`doc_type_rank` dedup, read-state regression tests); 295 tests, ruff/ty clean. |
| [skill-webapp-integration](docs/features/skill-webapp-integration/requirements.md) | **Shipped.** The skill-side counterpart to `synthesis-response-capture`, completing the clipboard→HTTP transition across both repos. The webapp gained a comment capture/read/integrate HTTP surface (`web/comments.py`: doc-id-keyed `POST /doc/{id}/comments` with replace-active-set + a `comment_submitted` events row + SSE broadcast; path-keyed `GET /comments` active-only with a `submitted` flag; `POST /comments/integrate` mark-by-id, doc-scoped), folding in the never-built `comment-capture` (#8) against the pre-existing empty `comments` table — no migration. The `doc-view` shell reads the iframe's `window.__fsComments` (a one-line backwards-compatible addition to the feature-skills spine templates, since a lexical `const` isn't a window property) and submits via a prominent bottom action-bar CTA; the footer "Copy" button was demoted to a secondary control. On the feature-skills side, all five `google-chrome` opens were dropped for the inbox, and the four review skills now poll `/synthesis-response` and fetch/integrate `/comments` over HTTP (force-walk → 5 s poll → clipboard fallback, defined once in `docs/webapp-polling.md`), retiring both clipboard round-trips. Five phase PRs across two repos (webapp #20–21, skills #1–4) + one direct-to-main review-fix round over two passes — the load-bearing fix: the skills were sending unexpanded `~` paths (and then literal `$HOME` inside single-quoted `-d` bodies) that never matched the walker's absolute `source_path`, silently 404ing the whole HTTP handoff. 255 tests, ruff/ty clean. |
| [synthesis-response-capture](docs/features/synthesis-response-capture/requirements.md) | **Shipped.** Answer a synthesis (feedback) doc in the webapp instead of the copy-to-clipboard / paste-into-chat round-trip. The walker now indexes feedback docs (which carry no `feature-doc-type` meta tag) via a filename-typing fork in `_process_file`, giving them a synthetic `<phase>-feedback` type. A new `web/synthesis.py` exposes `POST /doc/{id}/synthesis-response` (validate shape/integer-keys/string-values/1 MB before the transaction, atomic replace-on-submit into the pre-existing `synthesis_responses` table, broadcast on commit) and `GET /synthesis-response?path=<source_path>` (keyed by the dev-store path the agent already holds, with a `submitted` flag distinguishing unknown-doc from not-yet-answered). A fourth inbox category, "Awaiting your input" (active feedback docs with no submission yet), renders first and excludes those docs from "New since last visit"; submitting live-updates the open tab via the existing SSE broadcaster. Submission reads the same-origin, non-sandboxed feedback iframe's DOM directly from the `doc-view` shell — no `postMessage` bridge and zero feature-skills template change (a deliberate, signed-off divergence from the requirements); feedback docs are kept out of sibling-nav. The skill-side consumer of the read endpoint and clipboard retirement are deferred to `skill-integration-parallel`; the clipboard stays as a fallback until then. Four phase PRs (#16–19) + one direct-to-main review-fix round (reject non-string response values with 400); 215 tests, ruff/ty clean. |
| [sse-refresh](docs/features/sse-refresh/requirements.md) | **Shipped.** Live inbox updates via server-sent events — the open tab refreshes itself when docs land, change, or ship, no manual reload. A new in-process `web/broadcaster.py` holds one `asyncio.Queue` per connected tab; the single serialised walk worker signals it *from the event loop after the threaded walk commits* (the load-bearing cross-thread constraint), whenever a walk produced a net change. `WalkSummary` gained a `shipped` count and a `changed` gate so done-transitions broadcast on their own terms rather than riding an incidental `updated`. A `/events` endpoint (`sse-starlette` `EventSourceResponse`) emits a contentless `changed` message on connect and on every change, holds no DB connection, and deregisters each client in a `finally`. Phase 2 split the inbox card region into an `_inbox_body.html` partial served as a `?fragment=1` no-store, project-filter-preserving fragment, plus a vanilla `EventSource` client that refetches-and-swaps `#inbox-body` on any message (250 ms debounce; refetch-on-connect unifies live updates with stale-tab re-sync, `visibilitychange` as belt-and-braces — no HTMX). Two phase PRs (#14, #15) + one direct-to-main review-fix round; 177 tests, ruff/ty clean. |

Two phase PRs + one post-merge review-fix round (fold feature-id into the join, no-store test, dedup the Tracker literal); 156 tests, ruff/ty clean. |
| [inbox-view](docs/features/inbox-view/context.md) | **Shipped.** The headline UX — a cross-project inbox Home page replacing the placeholder at `/`. A read-model layer (`storage/inbox.py`: connection-taking `new_since_last_visit` / `in_progress` / `recently_shipped` + `build_inbox`, mirroring `read_state.py`) derives three categories: New since last visit (the unread predicate, INNER-JOINed to features so the null-feature tracker doc and missing/archived docs drop out, uncapped, newest-first), In progress (`features.status='in_progress'`), and Recently shipped (capped+30-day-windowed). Ship time comes from a new `shipped` event the walker emits on the done-transition (`document_id=NULL`, `{project,slug}` payload — matching the design's anticipated sse "ship" event) rather than the unreliable `features.updated_at`. Server-rendered Jinja, no HTMX; not-configured / all-empty / populated states (`Inbox.is_empty`); per-project filter chips; humanised type labels; display-only cards (click-through + read-stamp deferred to `doc-view`, "Awaiting your input" deferred to `synthesis-response-capture`).
